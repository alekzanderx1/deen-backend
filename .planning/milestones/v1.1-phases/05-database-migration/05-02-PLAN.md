  <action>
    Per D-04, run a targeted pgvector smoke test that proves the HNSW index is actually used. The basic `\dt` and `/_debug/db` checks would NOT catch a silent index misconfiguration where queries fall back to sequential scan.

    Pick the cleanest delivery (CONTEXT decisions allow Claude's discretion): a one-off Python script at `/tmp/05-02-pgvector-smoke.py` is fine and leaves no repo footprint. Do NOT add a permanent test file in `tests/` for this — it would require a Supabase connection in CI and that is out of scope.

    Script logic:
    1. Open a sync session via `db.session.SessionLocal()` (or `engine.begin()` if using Core).
    2. INSERT a row into `note_embeddings` with a real 1536-dim vector. Generate the vector with `[0.01] * 1536` or `numpy.random.rand(1536).tolist()`. Populate ALL non-null columns (read the model file to know which — likely `user_id`, `note_text`, `embedding`, plus any timestamps that don't auto-default). Use a sentinel `note_text` like `'pgvector smoke test 05-02 [timestamp]'` so you can find/clean it later.
    3. Run a cosine-similarity query against the same vector:
       ```sql
       SELECT id, embedding <=> :q AS distance
       FROM note_embeddings
       ORDER BY embedding <=> :q
       LIMIT 5;
       ```
       Confirm the inserted row comes back with a distance very close to 0.
    4. Run `EXPLAIN` on the same query and capture the plan text:
       ```sql
       EXPLAIN (FORMAT TEXT) SELECT id FROM note_embeddings ORDER BY embedding <=> :q LIMIT 5;
       ```
       Assert that the plan text contains the substring `Index Scan using` AND the index name from the migration (read it out of `alembic/versions/20260122_create_embedding_tables.py` — do NOT hardcode if uncertain). Assert it does NOT contain `Seq Scan on note_embeddings`.

       Note: HNSW indexes are only chosen by the planner when the query uses `ORDER BY ... <op> ... LIMIT k`. A bare `WHERE` filter will not trigger the index. The query above is shaped correctly.

    5. Cleanup: per D-04 Claude's discretion, DELETE the smoke test row by `note_text LIKE 'pgvector smoke test 05-02 %'`. Print the deleted row count.

    6. Save the EXPLAIN output to `/tmp/05-02-pgvector-explain.log` for the SUMMARY.

    Run the script:
    ```
    python /tmp/05-02-pgvector-smoke.py
    ```
    Expected exit code: 0. Expected stdout: `INSERT OK`, `SIMILARITY OK distance≈0`, `EXPLAIN OK uses Index Scan`, `CLEANUP OK 1 row`.

    If the EXPLAIN shows `Seq Scan` instead of `Index Scan`, this is a hard failure — surface it. It may mean the migration ran but the operator class is wrong, or the table has too few rows for the planner to bother with the index. In the latter case, set `SET enable_seqscan = OFF;` for the session and re-EXPLAIN — if it then uses the HNSW index, document it in SUMMARY as "planner chose seq scan due to small N; HNSW is functional under enable_seqscan=off".
  </action>
