The code is wired correctly: `ChatAnthropic` is used everywhere, `OPENAI_API_KEY` is gone from all changed files, `ModelConfig` enforces Claude constraints, and the AIMessage filter and `with_structured_output` patterns are in place. Test suite passes (10 classifier tests, 13 SEA tests).

**The single blocking gap is a runtime configuration mismatch:** The developer's `.env` file was not updated when Phase 9 was executed. It still contains:
- `LARGE_LLM=gpt-4.1-2025-04-14` (OpenAI model name)
- `SMALL_LLM=gpt-4.1-mini-2025-04-14` (OpenAI model name)
- `ANTHROPIC_API_KEY=sk-proj-...` (appears to be an OpenAI API key)

Because every `ChatAnthropic` factory function reads `LARGE_LLM`/`SMALL_LLM` from the environment (the lazy import pattern), these OpenAI model names flow directly into Anthropic's API at call time, causing the API to reject them as unknown models. The phase goal — "all LLM calls use ChatAnthropic; streaming fiqh response works end-to-end" — cannot be achieved until `.env` is updated.

The fix is purely operational (not a code change): update `.env` to match `.env.example` values for `LARGE_LLM`, `SMALL_LLM`, and `ANTHROPIC_API_KEY`.
